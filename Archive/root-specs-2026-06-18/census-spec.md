# census-spec.md

**Type:** script (`scripts/census.ts`) + CI drift check.
**Depends on:** `scripts/coverage-report.ts` (refactor to export its counters), `src/schema.ts`, `src/bankImport.ts`, the existing `npm run audit` / `.github/workflows/promotion-gate.yml` wiring.
**Concurrent-safe with:** content generation and renderer work; touches only `scripts/`, one new CI step, and the two bug fixes in §8.
**Status:** specced, not implemented.

Read `AGENTS.md`, `DECISIONS.md`, and `PROJECT-HISTORY.md` first; on any conflict they win. This spec operationalizes the "running census" open thread in `DECISIONS.md` and applies principle 3 (deterministic core; LLM only for the irreducible residual) to project documentation.

---

## 1. Purpose

The bank census — file counts, item-type / category / difficulty distributions, visual inventory, ID-uniqueness, target gaps — is pure counting, distribution, and set membership over `banks/*.json`. Under principle 3 it must be a script that returns identical verdicts every run, never a model call. The 2026-06-09 census was hand-run by Sonnet only because `PROJECT-HISTORY.md` had drifted from disk (it omitted `capnography-canonical.json`, which was bundled and validated). A script run costs ~0 tokens, is reproducible, and — wired into CI — cannot silently go stale.

This is a **reporter, not a verifier.** It describes the bank. It does not certify answer-placement integrity (that is `audit:positions`), reference hygiene (`audit:references`), or review status (the `BANK-REVIEW-LEDGER.md`). Keeping that boundary is what lets the census stay a fast, dependency-light counter.

## 2. What already exists — extend, do not replace

`scripts/coverage-report.ts` already computes most of this and prints Markdown to stdout:

- Loads every `banks/*.json` (plus argv extras), validates each via `validateBankObject`, and counts category / itemType / difficulty.
- A **kind-agnostic** `collectVisuals` that recurses into `case_study` exhibits, stages, and embedded questions — new visual kinds tally automatically.
- Rhythm sub-type counts, normalized-topic buckets (with each topic's categories + itemTypes).
- `normalizeTopic` (the single topic-normalization canon — reuse it, do not reimplement).
- **The target arithmetic is already here:** `categoryAverage = questions.length / categories.length`, the item-type equivalent, and the derived `PRIORITIZE_TOPICS` / `AVOID_TOPICS` lists. The §9 "target ~130 / ~148, under/over-served" content is this section, not new work.

The census is therefore `coverage-report` plus a structured emitter, a few cuts it does not make, and a drift check.

## 3. Architecture

1. **Refactor `coverage-report.ts` to export a pure `computeCoverage(questions): CoverageData`** that returns the structured object instead of `console.log`-ing. The existing CLI keeps working by calling `computeCoverage` and rendering Markdown from the result. This guarantees one normalizer and one set of counters feed both outputs — no producer/checker drift between two implementations (principle 2).
2. **`scripts/census.ts`** imports `computeCoverage`, adds the §5 cuts, and produces a `CensusData` object.
3. Emit **`census.json`** (the machine-readable source of truth) and render **`BANK-CENSUS.md`** from it. Stamp the Markdown with a generated-artifact header: `<!-- GENERATED by \`npm run census\`. Do not hand-edit. -->`.
4. Wire `"census": "tsx scripts/census.ts"` and `"census:check": "tsx scripts/census.ts --check"` into `package.json`.

## 4. `census.json` shape

```
{
  "generatedAt": "<ISO8601>",        // excluded from --check equality
  "gitSha": "<HEAD sha>",            // excluded from --check equality
  "perFile": [
    { "file", "schemaVersion", "metaCount", "questionsLength", "mismatch": bool }
  ],
  "totals": {
    "topLevel", "caseStudyTopLevel", "embeddedParts", "gradedTotal"
  },
  "byItemType": { ... }, "byCategory": { ... }, "byDifficulty": { ... },
  "bySchemaVersion": { "<ver>": { "questions", "files": [ ... ] } },
  "bySourceFile": { ... },
  "visuals": {
    "total", "byKind": { ... }, "rhythmSubtypes": { ... },
    "idsByKind": { "<kind>": [ "<questionId>", ... ] }
  },
  "caseStudies": [ { "id", "topic", "parts", "bank" } ],
  "idUniqueness": { "duplicates": [ { "id", "files": [ ... ] } ] },
  "targets": {
    "categoryAvg", "itemTypeAvg",
    "underCategories": [ ... ], "overCategories": [ ... ],
    "underItemTypes": [ ... ]
  },
  "prioritizeTopics": [ ... ], "avoidTopics": [ ... ],
  "docsDrift": { "ok": bool, "findings": [ ... ] }
}
```

## 5. Cuts to add beyond `coverage-report`

- **Per-file `meta.count` vs `questions.length`** — the §1 mismatch check. Flag any file where they differ.
- **Per-source-file and per-schema-version counts** — §3 "By source file" / "By schema version".
- **Top-level vs embedded split** — `topLevel`, `caseStudyTopLevel` (count of `itemType === "case_study"`), and `embeddedParts` (sum of `caseStudy.questions.length`), giving `gradedTotal`. `coverage-report` folds these together; the census separates them.
- **Cross-bank ID uniqueness** — collect every top-level `question.id` and every embedded `caseStudy.questions[].id` into one set; any collision is a finding. This is the regression guard for the "globally unique IDs" invariant; expected output is empty.
- **Visual ID enumeration by kind** — §4's per-kind ID lists, for spotting gaps (e.g. `lab_trend`/`mar` at 0).

## 6. Docs-drift check (`--check`)

Two layers, cheapest first:

1. **Self-consistency / staleness gate (authoritative).** Regenerate `census.json` and `BANK-CENSUS.md` in memory and compare to the committed files, ignoring `generatedAt` and `gitSha`. Any difference ⇒ exit non-zero with the diff. This is the principle-2 pattern: the checker re-runs the deterministic transform and asserts equality. It makes a stale committed census impossible to merge.
2. **Prose-drift warnings (best-effort).** Parse the count assertions out of `PROJECT-HISTORY.md` and `BANK-REVIEW-LEDGER.md` (e.g. the "Current canonical banks" list and any per-file totals) and warn — not fail — on any that disagree with `census.json`. Because prose format is soft, this layer warns and names the stale line rather than blocking; it would have caught the missing `capnography-canonical.json` entry. Keep it advisory so doc-wording changes never break CI.

## 7. CI wiring

Add a `census` step to `.github/workflows/promotion-gate.yml` that runs `npm run census:check`. Sequence it after `validate:bank` (no point counting an invalid bank) and alongside `npm run audit`. Net effect: every PR that changes a bank must also commit the regenerated census, or the gate fails.

## 8. Bugs to fix as part of this work

- **`coverage.json` / `--json` is a no-op.** `getBankFiles` treats all argv extras as bank paths, so `--json` is read as a filename. The committed `coverage.json` is just npm's captured preamble. Replace with the real emitter in §3 (`census.json`), and either delete `coverage.json` or have `coverage-report --json` print `CoverageData`.
- **`normalizeTopic` erases CJK — settled rule.** `replace(/[^a-z0-9\s/-]/g, " ")` strips all non-ASCII, so any Simplified-Chinese text in `topic` would collapse to spaces before counting. **Decision: `question.topic` is English-only.** Topics are navigational labels, not study content — the target user's working English covers them, while the translation does its real work in the stem, options, and rationale, where the medical terminology lives. Enforce this in `validateBankObject` (`src/schema.ts`), **not** in the census: every consumer (`coverage-report`, `census`, `validate:bank`, `promote`) already validates before counting, so a single Tier-0 rule rejecting CJK in `topic` propagates to all of them at once and fails loudly instead of silently stripping. The census normalizer's ASCII assumption then holds *by construction* — it never sees CJK, because validation already rejected it — so `normalizeTopic` is left unchanged. Bonus: the first gate run over the existing banks fails loudly if any current `topic` already contains Chinese, which surfaces the leak rather than hiding it.

## 9. The semantic residual (out of scope for the deterministic census)

The only parts of the 2026-06-09 census that needed judgment were the **near-duplicate case-topic adjudications** ("intentional diversity?") and the editorial "unless a novel angle exists" notes. Handle these the principle-3 way, *outside* `census.ts`:

- `scripts/find-duplicates.ts` already exists — use it (or a thin wrapper) to deterministically **flag** candidate near-dupes (shared category + itemType, normalized-topic proximity) and cap the list.
- A model adjudicates only the flagged, capped tail; cache each verdict keyed by the item-ID pair + a content hash so it is recomputed only when the content changes.
- The census merely links to this output; it does not embed judgment. The deterministic census stays 100% reproducible.

## 10. Acceptance / selfCheck

- `totals.gradedTotal === topLevel + embeddedParts`; per-category and per-item-type sums equal `topLevel`.
- `idUniqueness.duplicates` is empty on the current bank.
- Two consecutive runs produce byte-identical `census.json` and `BANK-CENSUS.md` modulo `generatedAt` / `gitSha`.
- `npm run census:check` exits 0 on a freshly generated census and non-zero after any hand-edit to a bank that is not reflected in a regenerated census.
- Cross-check once against the 2026-06-09 hand census: totals (1037 top-level, 155 embedded, 62 visuals) and the per-file table should match, modulo any bank changes landed since.
- Upstream contract: `validateBankObject` rejects any `topic` containing CJK (Tier 0). The census relies on this and does not re-check it; if validation is bypassed, the ASCII normalizer assumption no longer holds.

## 11. Non-goals

- Not a placement / reference / review verifier — that is `audit:*` and the ledger.
- Does not hand-edit `PROJECT-HISTORY.md`. The census *is* the count of record; PROJECT-HISTORY prose should eventually cite or embed the generated `BANK-CENSUS.md` rather than restate numbers that drift. (Hand-patching PROJECT-HISTORY's counts would repeat the original sin.)
