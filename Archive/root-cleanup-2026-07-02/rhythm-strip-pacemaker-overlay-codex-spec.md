## Decision Points — Resolved

The original handoff posed five open questions for adjudication before implementation. Resolved below after reading the live renderer (`src/visuals/kinds/rhythmStrip.ts`) and the closest existing precedent (`src/visuals/kinds/injection_site/index.ts`, the vessel-relation `selfCheck`).

**1. New `rhythmClass` values, a `pacer` overlay, or both → overlay only, no new `rhythmClass` values.**
The renderer builds its waveform by summing Gaussian bumps (`gaussian(t, center, sigma, amp)`) for P/QRS/T components into a per-sample `mv(t)` signal (see `beatMvAt`, `rhythmBaselineMv`, `atrialPWaveMv` inside `renderRhythmStripSvg`'s sampling loop). A pacing spike is the same primitive — a tall, narrow Gaussian bump — so it composes as one more term in that same sum, with `spec.rhythm` continuing to describe the *underlying* intrinsic rhythm (`asystole` for a fully pacer-dependent baseline, `sinus_brady` for failure-to-pace-during-bradycardia, `sinus`/`avb_3`/`avb_1` for failure-to-sense scenarios that need visible intrinsic activity for spikes to land on). This reuses the existing waveform machinery instead of forking per pacing scenario, and mirrors the injection-site precedent: a per-item behavioral distinction (capture vs. not, sense vs. not) is a data-contract field, not enum growth.

**2. Preserve IDs vs. retire + reissue → retire, reissue new IDs.**
Progress, flags, SRS, and answer history all key on `question.id` (`AGENTS.md`). Swapping a text item for a visual one under the same ID would silently carry her seen/missed/mastery state onto a materially different cognitive task. Retire the three old IDs from `visual-canonical.json`, ship new visual items with new IDs, record the replacement mapping (old ID → new ID, reason) in `BANK-REVIEW-LEDGER.md`.

**3. Stop at the 3 pacemaker candidates, or also pull in the 4 Bucket-2 candidates → stop at 3.**
Bucket 2 needs no architecture work, but `rhythm_strip` is already the largest visual kind (44 of 154 total visuals) while bowtie/highlight/case_study/fill_in_blank are the under-served item types. Treat Bucket 2 as a separate, explicitly-decided content lane later — not bundled into the architecture pass.

**4. Does the renderer have enough headroom for paced beats / QT visuals → yes, confirmed by direct read.**
`qtSec` already exists on `RhythmStripVisual` and already drives T-wave timing (`tSec: rSec + qtSec * 0.58` and similar), so the Bucket-2 QT-prolongation candidate (`ekg_b5_mc_03`) needs no renderer change — out of scope for this pass regardless, per #3. For paced beats: the existing `Beat` shape (`rSec`, `qrsSec`, `wide`, `rAmpMv`) already renders a visually distinct wide QRS — exactly what `pvc`/`vtach` use today — so a paced ventricular beat is "one more `Beat` pushed into the array with `wide: true`," not a new geometry primitive.

**5. Where smoke fixtures live → `audit/rhythm-strip-pacemaker-smoke-<date>/`.**
Matches the established convention (`audit/burn-map-smoke-2026-06-28-jaw-shoulders/`, `audit/burn-map-smoke-2026-06-27/`, etc.). Outside `banks/banks-raw/` and outside canonical — not treated as reviewed content per the handoff's "Phase 2 — Smoke Items Only."

## Architecture

### Data contract addition (`src/visuals/kinds/rhythmStrip.ts`)

```ts
export type PacerFinding = "capture" | "failure_to_capture" | "failure_to_sense" | "failure_to_pace";

pacer?: {
  mode: "ventricular";                 // Phase 1 scope only — see note below; do not widen without a rendering-semantics decision
  setRateBpm: number;                 // programmed pacing rate — defines expected spike cadence
  captureLatencySec?: number;         // default 0.08; delay from pacing spike to captured paced QRS
  spikeTimesSec: number[];            // every spike actually rendered
  capturedSpikeTimesSec: number[];    // subset of spikeTimesSec that produces a paced QRS
  finding: PacerFinding;              // declared scenario — selfCheck asserts the data backs this up
};
```

**Ventricular-only for Phase 1 (hardening, post-litigation):** the type is restricted to the literal `"ventricular"`, not a wider union with unused members. The renderer integration points below only ever define capture behavior for `mode: "ventricular"` (`wide: true` QRS synthesis) — `"atrial"`/`"dual_chamber"` have no defined rendering semantics anywhere in this spec, so leaving them in the type would let a validated spec pass with undefined render behavior. A literal-type restriction is enforced at compile time, which is stronger than a comment Codex has to remember; widen the union only when atrial/dual-chamber capture semantics are actually designed (a future spec, not a Phase 1 afterthought).

`finding` is declarative (same role as injection site's `vessel`), not derived — it states what the item is teaching, and `selfCheck` proves the numeric data actually demonstrates it. Mirror the injection-site pattern: add `meta.expected.pacerFinding` on the question and cross-check it against `pacer.finding` for redundant audit-readability, same as `meta.expected.route`.

### Renderer integration points (exact, in `rhythmStrip.ts`)

- **`buildIntrinsicBeats(spec, rng)`** (renamed from `buildBeats`, body otherwise unchanged — closing-hardening fix, a Codex catch): the existing per-rhythm logic stays exactly as-is, but the name now reflects what it actually computes — intrinsic beats only, with zero pacer awareness. This split matters because Phase 1 needs the intrinsic-only beat list available on its own: `selfCheck`'s `failure_to_pace`/`failure_to_sense` assertions need to reason about the intrinsic rhythm *independent of* whatever the pacer is doing, and if a single combined function both computed intrinsic beats *and* appended pacer-captured beats, calling that same function for self-check purposes would hand back a mixed list — corrupting exactly the check it's trying to perform (a captured paced beat sitting inside what should register as an intrinsic gap would wrongly appear to "fill" that gap, even though the pacer filling the gap is the point being tested, not evidence against `failure_to_pace`). Export `buildIntrinsicBeats`.
- **New `composeRenderBeats(spec, rng)`** — calls `buildIntrinsicBeats(spec, rng)`, then, if `spec.pacer` is present, appends one synthesized `Beat` per entry in `capturedSpikeTimesSec` (`rSec = spikeTime + captureLatencySec`, `wide: true` since Phase 1 is ventricular-only). Uncaptured spikes (in `spikeTimesSec` but not `capturedSpikeTimesSec`) get no `Beat` — the spike renders with nothing following it, which *is* the failure-to-capture finding. `renderRhythmStripSvg` calls this (not `buildIntrinsicBeats` directly) to get its render beat list; `selfCheck` calls `buildIntrinsicBeats` directly, never `composeRenderBeats`, so it only ever sees intrinsic timing.
- **Per-sample composition loop in `renderRhythmStripSvg`** — add a `pacerSpikeMv(timeSec, spec)` term to the existing `rhythmBaselineMv + atrialPWaveMv + Σ beatMvAt` sum. Implement as the same `gaussian()` helper already in the file. **Sigma floor to avoid sample aliasing (closing-hardening fix, a Codex catch):** the file samples every `sampleStepSec = 0.004`s, and "very small sigma" as originally specified risks a spike's Gaussian peak falling between samples, making it disappear or render with inconsistent height depending on exact phase alignment — directly undermining the *Visual subtlety* risk already named below ("do not ship if a novice can't see the cue"). Use a sigma floor of **0.008–0.012 seconds** (2–3× the sample step), which keeps the spike visually narrow relative to the ~6-second strip and comparable to or slightly sharper than an intrinsic QRS's own sigma (`qrsSec / 4.8` to `qrsSec / 7.5`, typically ~0.011–0.033s for `qrsSec` 0.08–0.16s) while guaranteeing several samples land inside the spike's visible width regardless of exact timing. Tall amplitude unchanged. If the Gaussian approach still looks inconsistent in practice after this floor, the fallback is rendering spikes as explicit SVG vertical marks rather than sampled sum terms — a bigger architectural change from Decision Point #1's "one more Gaussian term" choice, so only reach for it if the floored-sigma approach genuinely doesn't read clearly.
- **`validateRhythmStrip`** — extend with bounds checks for the new fields using the existing `bounded()` helper pattern: each `spikeTimesSec` entry within `[0, durationSec]`, `captureLatencySec` within `[0.03, 0.20]` seconds, normalized to a default of `0.08` when omitted (same `normalizeSpec`-style default pattern the file already uses for `durationSec`/`seed`/`calibrationPulse`). **Array-relationship invariants (hardening, post-litigation — a GPT pre-Codex catch, confirmed against the `bounded()` pattern and worth stating explicitly since none of this was in the original bounds list):**
  - `setRateBpm` finite and bounded (reuse `bounded()`, same shape as `rateBpm`'s existing check).
  - `spikeTimesSec` must be non-empty whenever `spec.pacer` is present — a pacer object with zero spikes is malformed, not a valid "no pacing" state (that's expressed by omitting `pacer` entirely).
  - `spikeTimesSec` entries must be unique — reject exact duplicates rather than silently de-duping (fail loud, matching this repo's JSON-hygiene precedent of surfacing malformed data with location rather than auto-correcting).
  - `capturedSpikeTimesSec` must be a **subset** of `spikeTimesSec` (every captured time also appears in the spike list) — not a *strict* subset: `finding: "capture"` requires full equality (every spike captured), so exact-set-equality must validate cleanly, not trip a strictness check.
  - For every entry in `capturedSpikeTimesSec`: `spikeTime + captureLatencySec <= durationSec`, so a captured beat's synthesized QRS never renders past the strip edge.
- **`rhythmStripModule` export** — currently has no `selfCheck` at all; this is a genuinely new addition, not a modification. Model it directly on `selfCheckInjectionSite` in `src/visuals/kinds/injection_site/index.ts`: read `question.meta.expected.pacerFinding`, cross-check against `spec.pacer.finding`, and assert the numeric data is internally consistent with the declared finding (see *Self-Check Assertions* below).
- **`allowedKeys.ts` / schema doc (closing-hardening fix, a Codex catch, confirmed by direct read):** `allowedKeySets.visualByKind.rhythm_strip` in `src/allowedKeys.ts` currently lists `rhythm`, `rateBpm`, `durationSec`, `seed`, `calibrationPulse`, `atrialRateBpm`, `conductionRatio`, `prSec`, `qrsSec`, `qtSec` — **no `pacer`**. Without adding it, `validateBankObject({ rejectUnknownKeys: true })` (which both `npm run validate-bank` and `consolidate.ts` run unconditionally) rejects every pacer visual as an unknown key, independent of whether `validateRhythmStrip`'s own shape checks pass. Add `"pacer"` to that array. Also update `NCLEX-Question-Schema.md`, which currently documents rhythm_strip as having no `selfCheck` at all — that line goes stale the moment Phase 1 lands and should be corrected in the same pass, not left for someone to notice later.

**Shared-derivation requirement:** whatever function computes "where are this rhythm's intrinsic beats" for `failure_to_sense` / `failure_to_pace` checking must be the *same* function used to render them — `buildIntrinsicBeats`, called directly by `selfCheck`, never a second parallel derivation that could drift from what's actually drawn. This is principle 2/11 (producer ≠ checker only via a shared deterministic transform; single definition, never redefined) applied to this kind.

**RNG-determinism note — corrected (closing-hardening fix, a Codex catch against the litigation pass's own prior note, which was wrong):** the prior version of this note said the self-check should "reconstruct `mulberry32(spec.seed)` fresh... and call the exported `buildBeats` with that fresh instance," claiming this reproduces bit-identical output. **That's incomplete and wrong for `afib` specifically.** Confirmed directly against `renderRhythmStripSvg`'s actual call order:
```ts
const rng = mulberry32(spec.seed);
const rngValues = [rng(), rng(), rng(), rng()];   // 4 calls consumed here, for baseline noise (vfib f1/f2/f3, one reserved)
...
const beats = buildIntrinsicBeats(spec, rng);      // rng is already 4 calls advanced by this point
```
`buildIrregularRPeaks` (used for `afib`'s jittered intrinsic timing) consumes further calls from that *already-advanced* `rng`. A self-check that constructs a truly fresh `mulberry32(spec.seed)` and hands it straight to `buildIntrinsicBeats` — skipping the 4-value consumption — starts `afib`'s jitter sequence from a different point in the stream than what actually rendered, silently desynchronizing self-check from render on exactly the rhythm where jitter matters most.

**Fix:** export a small shared render-context builder, e.g. `buildRenderContext(spec: NormalizedRhythmStripVisual): { rng: Rng; rngValues: readonly number[] }`, that does exactly the `mulberry32(spec.seed)` + 4-value-consumption step and nothing else. `renderRhythmStripSvg` calls it once and uses the returned `rng` for `buildIntrinsicBeats`/`composeRenderBeats` and `rngValues` for baseline noise. `selfCheck` calls the *same* `buildRenderContext(spec)` and feeds its `rng` into `buildIntrinsicBeats` directly. Neither side needs to know or remember "4 values, in this order" as an implementation detail to replicate — that detail lives in exactly one place, and both call sites consume it the same way. This is a cleaner instance of the same shared-derivation principle above, not a new one: the bug wasn't in the *principle*, it was in the *previous instruction's* incomplete implementation of it.

### Self-Check Assertions

**Non-pacer items are a hard no-op (hardening, post-litigation):** `if (spec.pacer === undefined) return [];` as the very first line, before any other check. `rhythm_strip` has 44 existing non-pacer canonical items today and this is the kind's *first* `selfCheck` ever — it must never require anything of an item that has no `pacer` field, or every existing item fails a check that didn't exist when it was authored.

When `spec.pacer` is present:

- `meta.expected.pacerFinding` is **required**, not merely compared when present (hardening, post-litigation — a GPT pre-Codex catch, confirmed against the actual precedent: `selfCheckInjectionSite` requires `meta.expected.route`, the primary declared finding, and only treats the secondary `meta.expected.target` as an optional cross-check. `pacerFinding` is `route`'s equivalent here — the primary declared thing the whole self-check exists to verify — so it follows the required pattern, not the optional one). Missing it on a pacer item is itself a self-check failure; when present it must equal `spec.pacer.finding`.
- `finding: "capture"` → `capturedSpikeTimesSec.length === spikeTimesSec.length` (every spike captured).
- `finding: "failure_to_capture"` → `capturedSpikeTimesSec.length < spikeTimesSec.length` (at least one uncaptured spike). This is the handoff's own suggested test: a declared failure-to-capture with all spikes captured must fail.
- `finding: "failure_to_pace"` → requires the base rhythm's intrinsic-beat gaps (from the shared derivation above) to contain at least one interval longer than `60 / setRateBpm` with no spike inside it. **Gaps must include the strip edges (hardening, post-litigation — a GPT pre-Codex catch, and the highest-value one: confirmed directly against `buildIntrinsicBeats` that `rhythm: "asystole"` and `rhythm: "vfib"` both return zero intrinsic beats — `if (spec.rhythm === "vfib" || spec.rhythm === "asystole") return beats;` with `beats` still empty — and Decision Point #1 explicitly names `asystole` as the intended underlying rhythm for a fully pacer-dependent baseline. A "gaps between consecutive beats" check alone has no adjacent pairs to compare when there are zero or one intrinsic beats, so it would silently pass a `failure_to_pace` item with garbage or absent spike data in exactly the scenario this spec calls out as the primary use case.** Concretely, the gap set to check is: `[0, firstIntrinsicBeat]`, each `[beat[i], beat[i+1]]`, `[lastIntrinsicBeat, durationSec]`, and — when there are zero intrinsic beats — the whole strip `[0, durationSec]` as a single gap.
- `finding: "failure_to_sense"` → requires at least one entry in `spikeTimesSec` to land within a small epsilon window of an intrinsic beat's QRS/T window from the base rhythm.

The `failure_to_pace` and `failure_to_sense` assertions depend on `buildIntrinsicBeats`/`buildRenderContext` being exported and used correctly (see *Shared-derivation requirement* and *RNG-determinism note* above) — treat getting that export right as the core Phase 1 technical risk, not the spike-drawing itself.

### Schema-version policy (closing-hardening addition, a Codex catch — corrected in a second Codex pass, see status line)

This repo bumps `schemaVersion` deliberately for new optional structure on an existing kind — bowtie shipped at 1.4, `rationale.visuals` at 1.5 — so `pacer` follows the same pattern rather than being treated as a free-floating optional field with no version story. Current state, confirmed by direct read (not assumed): `SCHEMA_VERSION`/`supportedSchemaVersions` in `src/schema.ts` top out at `"1.6"`; **`banks/visual-canonical.json`'s own `meta.schemaVersion` is `"1.2"`** — notably lower than the `"1.6"` the three model-origin banks (`gpt-`/`claude-`/`gemini-canonical.json`) declare.

**How schema-floor enforcement actually works in production (corrected — verified line-by-line against `schema.ts`, not assumed from the registry mechanism's own docstring):** `validateVisual` accepts an `options.schemaVersion` and, when present, compares it against `mod.requiredSchemaVersion` — but neither production call site ever passes it. `validateQuestion` calls `validateVisual(raw.visual, "visual", reasons, { itemType, question })` — no `schemaVersion` key. `validateCaseStudyExhibit` calls `validateVisual(value.visual, path, reasons)` — no options object at all. `validateVisual`'s own docstring confirms this is intentional: *"`schemaVersion` is accepted for the registry-mechanics test; production schema-floor enforcement stays at the bank level in `validateBankObject`."* Production instead enforces every existing schema floor via **hardcoded, per-feature boolean checks inside `validateBankObject`'s question loop** — `schemaVersion === "1.0" && itemType === "case_study"` requires 1.1; `highlight` requires 1.3; `bowtie` requires 1.4; `hasRationaleVisuals(question)` requires 1.5; **any** visual (question-level or case exhibit/stage-exhibit) requires 1.2, generically, not per-kind; `hasSchema16CaseFields(question)` requires 1.6. There is no existing per-visual-*kind* floor check anywhere in this list — `mod.requiredSchemaVersion` is real registry plumbing, but nothing in the production validation path reads it today.

**Corrected resolution:**
1. Add `"1.7"` to `supportedSchemaVersions` in `src/schema.ts` and to the `SchemaVersion` union in `src/types.ts` — **reconfirm the current max at implementation time**, not from this spec, since other in-flight parallel specs may have already claimed 1.7 (multiple Codex instances are working this repo concurrently).
2. **Add a new `hasPacerRhythmStrip(question: Question): boolean` helper in `schema.ts`, modeled directly on the existing `hasSchema16CaseFields`**, and a matching hardcoded check in `validateBankObject`'s question loop (`schemaVersion !== undefined && cmpSchema(schemaVersion, "1.7") < 0 && hasPacerRhythmStrip(result.value)` → push a `"pacer rhythm_strip requires meta.schemaVersion 1.7"` reason), inserted alongside the existing bowtie/highlight/`rationale.visuals`/1.6-case-fields checks. `hasPacerRhythmStrip` walks: `question.visual` (kind `rhythm_strip` with `.pacer` present); for `case_study`, also `caseStudy.exhibits[].visual`, `caseStudy.stages[].exhibits[].visual`, and each embedded `caseStudy.questions[].visual` — the same shape the existing 1.2-generic-visual check already walks, just narrowed to the pacer case. This is the actual, live-enforced floor; setting `mod.requiredSchemaVersion` alone (harmless to also do, for consistency with the registry mechanism and its test) does **not** by itself protect a real bank from an under-versioned pacer item.
3. **`visual-canonical.json`'s own `meta.schemaVersion` must bump from `"1.2"` to `"1.7"` in the same deliberate commit that lands the first pacer item there** (Phase 3, not Phase 1) — needed regardless of enforcement mechanism, since `hasPacerRhythmStrip` will otherwise correctly reject the pacer item against the bank's stale declared version. This isn't a workaround — it's precisely the case `consolidate.ts`'s own version-rank guard is built for (`"bump the canonical deliberately before consolidating"`).
4. **Phase 1 is unaffected by any of this.** Fixture-based testing (`fixtures.valid`/`fixtures.invalid` in the module, plus `scripts/tests/rhythm-strip.ts`) exercises `validate()`/`selfCheck()` directly against spec objects, never through `validateBankObject`'s bank-envelope-level checks at all — so Phase 1 can proceed and pass its acceptance gates without touching `schema.ts`'s question-loop logic. The `hasPacerRhythmStrip` check only matters once Phase 3 actually promotes a pacer item into a canonical bank.

### Content Rewrite Rule (carried from the original handoff, unchanged)

The visual must carry the finding; the stem must not state it. `ekg_b5_mc_04`'s current stem ("several spikes are not followed by a QRS complex") is exactly the bad pattern and must be rewritten before becoming a visual item — e.g. "The nurse reviews the telemetry strip from a client with a ventricular pacemaker. ### Canonical replacement/edit mechanism (closing-hardening addition, a Codex catch)

Decision Point #2 says "retire the three old IDs... ship new visual items with new IDs" without specifying the mechanics, and the addendum's *Process* section leaves open whether Bucket 1B conversions reuse the same ID or issue a new one. Confirmed directly against `scripts/consolidate.ts`: it is **strictly append-only** — it merges staged questions into the canonical bank via `questions: [...canonicalBank.questions, ...staging.questions]` and explicitly **fails** (`ok: false`, "N duplicate ID(s) would result") on any ID collision between staging and canonical. It has no removal or in-place-edit capability at all.

This means two of this spec's own planned operations fall entirely outside the normal raw → promote → consolidate pipeline, and need to be named as such rather than assumed to "just work" through staging:

- **Retiring an old ID (pacemaker Phase 3):** removing `ekg_b5_mc_04`/`ekg_b5_mc_05`/`ekg_b5_matrix_10` from `visual-canonical.json` is a deletion. `consolidate` can't do it — staging the new pacer-visual replacements as raw drafts and promoting them normally is fine (they get new IDs, no collision), but the old entries' removal from the canonical file is a **separate, deliberate, manual edit** to `visual-canonical.json`, done alongside (not through) that promotion, with the `BANK-REVIEW-LEDGER.md` entry as the audit trail Decision Point #2 already calls for.
- **Bucket 1B conversions — always a manual edit, not just the same-ID case (corrected in a second Codex pass):** the first version of this note said only *same-ID* conversions were in-place edits, implying a new-ID conversion could go through staging. That's not quite right for any of the 4 placement-clean candidates, because of *what* is being modified, not just which ID it keeps:
  - The **two exhibit-based conversions** (`cs_adhf_pulm_edema_01`/`ed_assessment`, `gpt_stroke_..._warfarin_01`/`baseline_assessment`) add a `.visual` field to an exhibit **inside an existing case_study's canonical entry**. An exhibit has no independent top-level ID to reissue — there is no such thing as "stage this exhibit as new content"; the only unit `consolidate` can add or reject is a whole top-level question, and these cases already exist in canonical. This is unconditionally a same-entry edit, with no staging path available regardless of ID strategy.
  - The **two embedded-leaf conversions** (`opus26_case_refeeding_syndrome_01` leaves `_q3`/`_q5`) sit inside the same parent case_study's `caseStudy.questions` array. Whether the *leaf* keeps its existing ID or gets a new one (a real, open choice per the addendum's *Process* section, affecting progress/SRS semantics for that leaf) does **not** change the mechanical path: either way, the edit happens by modifying the **parent case_study's existing canonical entry** in place, because a case_study is one atomic top-level JSON object — you cannot stage a change to one embedded leaf as a separate new top-level draft without staging the entire parent case, which would itself collide with the existing parent ID in canonical. So even a "new ID for the leaf" choice still routes through a manual parent-entry edit, not `consolidate`.

  All 4 Bucket 1B v1 candidates are therefore **manual, deliberate, ledger-logged edits to an existing canonical entry**, human-reviewed before they land, exactly like the pacemaker retirement case above — never staged content promoted through `consolidate`.

Neither of these is a gap in the pipeline design — the pipeline is deliberately append-only so raw-generation mistakes can't silently overwrite reviewed canonical content (principle 5: generated ≠ reviewed). It does mean "retire," "same-ID edit," and "exhibit/embedded-leaf edit" are all **manual, reviewed, ledger-logged canonical edits**, explicitly outside `npm run consolidate`, not something Codex should try to route through staging.

## Addendum (2026-06-30, post-litigation): Narration-Debt Conversion Pass ("Bucket 1B")

Luke's framing, litigated and adjudicated: convert any item whose stem narrates a rhythm/EKG *shape* — not just names a diagnosis — into a visual item, even when the rhythm isn't the item's literal graded decision point. This is a distinct, independently-sequenceable workstream from the pacemaker overlay above: it needs **zero renderer changes** (every candidate already uses one of the 14 existing `RhythmClass` values), so it can run before, after, or in parallel with Phases 1–4.

### Why this is compliant with the decorative-visual rule, not an exception to it

Principle 6 ("a visual whose removal leaves the answer unchanged is decorative and invalid") stays intact *only if every conversion includes a stem rewrite that removes the narrated finding*. Adding a strip next to an unchanged stem would be genuinely decorative — the same failure mode the existing rule exists to catch. The visual must become load-bearing for at least the recognition step, exactly the same discipline already required for the 3 pacemaker items above (*Content Rewrite Rule*).

### The line: shape vs. label

A bare diagnosis name ("client with atrial fibrillation," used as PMH context for an unrelated pharmacology decision) has nothing to mentally construct — converting it would mean inventing rate/morphology detail not present in the original item, which is fabrication, not backfill. A narrated *shape* — irregularity, a rate tied to an event, wide/bizarre complexes, absent P waves, frequency of an ectopic beat — is the actual "I have to visualize this" trigger and is what qualifies.
