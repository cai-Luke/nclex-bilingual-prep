# AGENTS.md

Guidance for coding and content agents working on Project Shrimp / NCLEX Bilingual Prep.

## Project Coordinates

- Git repo: `https://github.com/cai-Luke/nclex-bilingual-prep.git`
- Published page: `https://cai-Luke.github.io/nclex-bilingual-prep/`
- App type: static offline Vite + React + TypeScript single-page app.
- Target user: an NCLEX-RN learner who benefits from bilingual English/Chinese scaffolding.

## Read First

- `PROJECT-HISTORY.md` is the current status map. Update it when a meaningful implementation pass lands, verification changes, or active scope moves.
- `NCLEX-Question-Schema.md` is the schema source of truth.
- `BANK-REVIEW-LEDGER.md` tracks content-review status and should be updated before generated questions are treated as reviewed study material.
- `NCLEX-Prep-SPEC.md` is useful product context, but it started as a build spec and may describe historical plans.
- `PASS2-HANDOFF.md` is historical.

## Product Constraints

- Keep the runtime static and offline. No server, API key, or live model call is required after build.
- English is the primary exam surface. Chinese is a learner scaffold and should remain toggleable.
- The app intentionally avoids readiness/pass-fail claims. Adaptive exam-condition practice exists for pacing and review, not certification.
- Do not add a persistent AI warning footer back into the study UI unless the product decision is explicitly revisited.

## Implementation Notes

- The app uses plain CSS in `src/styles.css`; Tailwind is not installed.
- Preserve the production build's `file://` compatibility path. `npm run build` runs `scripts/make-file-build.ts` after Vite build.
- Bundled banks are loaded from top-level `banks/*.json` via `src/banks.ts`.
- Nested bank files are not bundled or scanned by the default coverage script unless code is deliberately changed.
- Bundled source labels come from filename stems.
- Question IDs must stay globally unique across bundled top-level banks. Progress, flags, sessions, and answer history key by `question.id`.
- Uploaded-import collisions are regenerated by `src/bankImport.ts`, but bundled banks should not rely on that escape hatch.

## Question Bank Workflow

- Raw generated questions are not trusted study material by default.
- Put raw model output under `banks/banks-raw/` or another explicit raw/staging area, not directly into bundled top-level banks.
- Validate, audit, and source-check generated content before promoting it into `banks/*.json`.
- Record review status in `BANK-REVIEW-LEDGER.md`.
- For clinical claims, prefer authoritative sources such as professional guidelines, government health agencies, drug labels, or established clinical references.
- Be especially strict with medication, lab, dosage, prioritization, and delegation items.
- Do not AI-generate medical images. If image-dependent items are ever added, use curated licensed real images or deterministic data-derived visuals with human verification.

## Commands

Run these before calling a code/content pass complete when relevant:

```sh
npm run validate-bank -- banks/*.json
npm run coverage-report
npm run build
```

Useful development commands:

```sh
npm run dev
npm run preview
```

## Current Work Shape

The project is in late-stage maintenance and expansion:

- Polish existing UX without changing the offline/static architecture.
- Add more reviewed questions to under-covered topics and item types.
- Keep schema changes rare and deliberate.
- Add focused regression tests or browser automation if the tooling becomes available.
- Consider optional remote bank updates only if manual bundled-bank updates become annoying.

## Working Style

- Prefer small, well-scoped changes that match existing patterns.
- Do not rewrite app architecture during polish work.
- Do not treat generated question content as equivalent to reviewed content.
- When changing storage, grading, schema validation, or bank loading, verify broadly.
- If the worktree contains unrelated edits, leave them alone.
